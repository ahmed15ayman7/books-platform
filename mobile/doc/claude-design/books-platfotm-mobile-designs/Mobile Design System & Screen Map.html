Reading the room

What I found & what I'm assuming

Before tokens, here's the reasoning — flagging where the codebase contradicts the older written brief so you can correct me early.

⚑ The palette is red + black, not navy
The earlier written style guide described a navy/indigo (#46467F) brand. The shipped code tells a different, authoritative story: the live product is a "Soft UI" system in cultural red #B11E2E, near-black #0B0B0B and white over warm grays — explicitly commented أحمر + أسود + أبيض in globals.css. The prompt asks me to confirm hex from the code, so I'm treating the codebase as the truth and discarding the navy direction. Say the word if the brand has since moved back to navy.
⚑ "Soft UI" — rounder & springier than the web guide implied
The code is deliberately soft: large radii (cards 1.75rem, buttons 1.5rem), gentle navy-free shadows, and a spring easing cubic-bezier(.34,1.45,.64,1) on hovers/taps. I've carried that personality into the mobile foundation rather than the "scholarly, 6–8px corners" the old guide described.
⚑ Mobile is a separate sprint — this is the spec for it
Your answers note the mobile app is "a new separated scope sprint." This document is exactly that hand-off spec: it defines the foundation + screen map the mobile sprint builds against. No screens are designed yet — per the brief, this is for approval first.

Scope & business rules

Folding the clarification answers + the Prisma schema into a single source of decisions.

✓ In this version
  • E-commerce / Cart — activated. Checkout + a payment gateway (provider TBD).
  • Mixed buying — some books bought directly, some referred to the publisher.
  • Comments & ratings — on books & articles (admin-moderated).
  • Wishlist — save-for-later (email-keyed, no login).
  • Newsletter + push/Telegram notifications.
  • Publish your book — first book free, then paid; admin approval required.
  • B2B subscriptions, ambassador referrals, sponsored publishers.
✕ Out / deferred this version
  • Reader accounts / logins — not in this version. Cart & wishlist run on session + email, not auth.
  • Translated-book purchase links — informational only.
  • Self-serve publisher profiles — admin-created only for now.
  • Chosen payment provider — not selected; design must stay provider-agnostic.

Roles present in the DB (Admin/Editorial, Author, Publisher, Translator, Ambassador) match the previous system — surfaced through admin, not a public reader login.

Phase 2 · Plan

Open questions before screen design

Most ambiguities are resolved by your answers (folded into Scope above). These remain genuinely open and block specific screens:

1 · Payment gateway not chosen
Checkout is in-scope but the provider is undecided. I'll design checkout provider-agnostic (generic card + redirect-to-gateway pattern). Confirm target — local (e.g. region card processor) vs international (Stripe/PayPal) — as it changes which fields and trust marks appear.
2 · Direct-buy vs referral per book
Buying is "mixed." Confirm the rule that decides which a book shows — is it driven by purchaseOption / referralLink in the DB (my assumption), or an editorial toggle?
3 · Comments & ratings — gating without logins
Readers have no accounts this version, but comments/ratings are in. Assumption: email + name per comment (matches the schema), admin-moderated. Confirm this is acceptable vs. requiring a light verification.
4 · Dark mode at launch?
The code defines a full dark theme. Is dark mode a launch requirement for mobile, or phase 2? It affects how many specs each screen needs.
5 · Navy vs red — final brand confirmation
I went with the codebase red. One-line confirm and I lock it; if navy is the real current brand I'll re-issue the color section only.
Next step
On approval of these tokens + screen map (and answers to the 5 above), I'll design the 5 priority screens first — RTL-first, light mode — as a clickable Flutter-faithful prototype.
