## What Was Done

- Investigated user report: Android article TTS always showed "الصوت العربي غير متاح على هذا الجهاز..." while book TTS worked in Arabic AND English on the same device, same session. iOS worked everywhere.
- **Discovery 1:** articles were NOT using the shared `TtsPlayerWidget` — a diverged copy `ArticleTtsPlayerWidget` existed (`lib/features/articles/presentation/widgets/article_tts_player_widget.dart`) that force-selected a voice via `setVoice()` (pre-scanned from `getVoices` in `initState`) instead of `setLanguage`-first. On Android, `getVoices` returns only `name`+`locale` (no `network_required` key), so its "prefer offline voice" filter was a no-op — it picked arbitrary (often network/uninstalled) voices; `setVoice` reported success, `speak()` failed async, error handler permanently locked the error card.
- **Fix 1 (merge):** deleted the diverged copy + its now-empty `widgets/` folder; article call site (`article_detail_body.dart:83`) now uses shared `TtsPlayerWidget` with identical params (`languageCode: 'ar-SA'`). Added first widget test with mocked `flutter_tts` MethodChannel.
- **Fix 1 did NOT resolve it** — but isolated the real variable: both screens now ran identical code; the only difference left was **text length** (articles ~8,500 chars vs book descriptions ~300–800).
- User confirmed on device: **a short article played fine on Android** → root cause confirmed as Android's `TextToSpeech.getMaxSpeechInputLength()` (~4,000 chars per `speak()` call). Long text fails; the widget's `setErrorHandler` misreported any failure as voice-unavailable. iOS's `AVSpeechSynthesizer` has no such limit.
- Verified critical plugin semantics against `flutter_tts` 4.2.5 source in pub cache (`~/.pub-cache/hosted/pub.dev/flutter_tts-4.2.5/`):
  - `getMaxSpeechInputLength` is Android-only; iOS returns `FlutterMethodNotImplemented` → throws in Dart (must platform-guard).
  - With `awaitSpeakCompletion(true)`: awaited `speak()` resolves **1** on natural completion, **0** on stop/pause/error — a chunk loop can't hang.
  - `setCompletionHandler` fires per utterance — would flicker `_isPlaying` between chunks; removed.
- **Fix 2 (chunking):** implemented sentence-boundary text chunking with sequential playback in the shared widget.
- `flutter analyze` → No issues found. New tests 13/13 pass. Full suite 97 pass + only the 5 known pre-existing failures (verified pre-existing by stash + re-run on clean HEAD).
- **User verified on real devices: TTS now works correctly on BOTH iOS and Android, on BOTH books and articles (including long articles).** ✅
- Updated auto-memory (`project_tts_feature.md`) with the platform gotchas.

## Notes

- **Do NOT reintroduce per-feature TTS widget copies** — the divergence itself was bug #1's vector.
- Chunk seams: a barely perceptible pause may occur between chunks on Android (~every 4,000 chars); sentence-boundary splitting keeps it natural. Inherent to the platform limit.
- `_playSession` counter prevents a stale chunk loop from resurrecting playback state after fast pause→play.
- The 100-char safety margin under `getMaxSpeechInputLength` is deliberate (OEM engines can misbehave near the exact limit).
- The `- 100` result is guarded (`> 0 ? maxLength : 3900`) against absurd engine values.
- Timeline addendum: Jul 2 — Android TTS bugs fixed (merge + chunking), verified working on both platforms by user.
