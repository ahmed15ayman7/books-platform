### `firebase_options.dart` was created manually

The gem was then installed (`gem install xcodeproj`), but the Dart file was created manually by reading the already-generated `google-services.json` and `GoogleService-Info.plist` and extracting the values. The values are correct and identical to what the CLI would have generated.

### Backend endpoint naming difference from handoff 004

Handoff 004 planned `/notifications/mobile/register` (POST) and `/notifications/mobile/unregister` (POST). The actual backend uses a single endpoint at `/notifications/mobile/subscribe` — **POST to register, DELETE to unregister**. The mobile data source was written to match the actual backend.

### `FcmToken` model (not `DeviceToken`)

Handoff 004 planned to create a `DeviceToken` model. The actual Prisma model is named `FcmToken` and has additional fields: `topics[]`, `isActive`, `lastUsedAt`, and optional `userId` (mobile users are anonymous — `userId` stays null for app token registrations).

### `shouldBoostPosition` is the correct publish trigger on PATCH

In `web/app/api/v1/admin/books/[id]/route.ts`, the flag `shouldBoostPosition` is `true` exactly when a book transitions from unpublished to published (`parsed.data.published === true && (!existing.published || existing.position === 0)`). This is used as the FCM trigger on update so notifications only fire on the actual publish event, not on every book edit.

### Android build files are Kotlin DSL (`.kts`)

The handoff 004 plan had Groovy syntax for Gradle changes. This project uses Kotlin DSL (`.kts` extension). The changes applied this session use correct Kotlin DSL syntax.
